简介

Git 是用于 Linux 内核开发的版本控制工具。与常用的版本控制工具 CVS, Subversion 等不同，它采用了分布式版本库的方式，不必服务器端软件支持（这得分是用什么样的服务端，使用 http 协议或者 git 协议等不太一样。并且在 push 和 pull 的时候和服务器端还是有交互的。），使源代码的发布和交流极其方便。Git 的速度很快，这对于诸如 Linux kernel 这样的大项目来说自然很重要。 Git 最为出色的是它的合并跟踪（merge tracing）能力。

对于任何一个文件,在 Git 内都只有三种状态:

  • 已提交(committed) -- 已提交表示该文件已经被安全地保存在本地数据库中了
  • 已修改(modified) -- 已修改表示修改了某个文件,但还没有提交保存
  • 已暂存(staged) -- 已暂存表示把已修改的文件放在下次提交时要保存的清单中

Git 安装完成后先配置账户和邮箱

$ git config --global user.name "jiwanger"
$ git config --global user.email "jiwanger@126.com"

Git 分支管理策略

在实际开发中，我们应该按照几个基本原则进行分支管理：

首先，master 分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；

那在哪干活呢？干活都在 dev 分支上，也就是说，dev 分支是不稳定的，到某个时候，比如 1.0 版本发布时，再把 dev 分支合并到 master 上，在 master 分支发布 1.0 版本；

你和你的小伙伴们每个人都在 dev 分支上干活，每个人都有自己的分支，时不时地往 dev 分支上合并就可以了。

初始化一个git仓库

  • 第一种是在现存的目录下,通过导入所有文件来创建新的 Git 仓库。

    $ git init
$ git add .
$ git commit -m "initlize"

  • 第二种是从已有的 Git 仓库克隆出一个新的镜像仓库来。

    $ git clone https://www.github.com/jiwanger/note.git

gitignore 文件配置

文件 .gitignore 的格式规范如下:

  • 所有空行或者以注释符号 # 开头的行都会被 Git 忽略。
  • 可以使用标准的 glob 模式匹配。
  • 匹配模式最后跟反斜杠( / )说明要忽略的是目录。
  • 要忽略指定模式以外的文件或目录,可以在模式前加上惊叹号( ! )取反。 
# 此为注释 – 将被 Git 忽略
*~			# 忽略所有 ~ 结尾的文件, 如 vim 的副本文件就是这样
.settings/	# eclipse 的项目配置目录
.project	# eclipse .project文件
*.a			# 忽略所有 .a 结尾的文件
!lib.a		# 但 lib.a 除外
/TODO		# 仅仅忽略项目根目录下的 TODO 文件,不包括 subdir/TODO
build/		# 忽略 build/ 目录下的所有文件
doc/*.txt	# 会忽略 doc/notes.txt 但不包括 doc/server/arch.txt

对于 .gitignore 中忽略的文件：

$ git add -f App.class			# 就算 .gitignore 忽略了该文件，也可以添加进去
$ git check-ignore -v App.class # 此命令可以检查哪个规则忽略了改文件

常用命令

以 version.txt 文件为例

1. 添加/提交/状态/日志/比对

## 1.version.txt 内容：
## ONE: ONE: The first line.
## TWO: The second line. 
$ git add .                   # 跟踪新文件/目录, 暂存已修改文件
$ git commit -m 'init'        # 提交更新到库
$ git status                  # 检查当前文件状态(是否有更改)

## 2.version.txt 内容：（第一行添加: number）
## ONE: The first number line.
## TWO: The second line. 
$ git commit -am 'add number'  # 暂存已修改文件并更新
$ git log                      # 查看前面两次提交

## 3.version.txt 内容：（第二行: second -> third）
## ONE: The first number line.
## TWO: The third line. 
$ git status				   # 查看状态（提示有文件修改未暂存）
$ git diff version.txt         # 比对修改的文件
$ git add .                    # 暂存所有文件
$ git status                   # 查看状态（提示有文件提交）
$ git commit -m 'modify third' # 提交

2. 版本回退

$ git log --pretty=oneline     # 显示从最近到最远的提交日志(单行模式显示)
## 2fa86e3d1416ddcfca369ccec5ef8392156e3256 modify third
## d256c95bd173362053eb04bea4ff91a5865fc8f2 add number
## a2989999cd2f7aac717fde3b5f7354f3c9deba6d first commit

$ git reset --hard d256c95bd173362053eb04bea4ff91a5865fc8f2
## version.txt 的内容变为第二次提交的内容
## ONE: The first number line.
## TWO: The second line.

$ git log --pretty=oneline     # 此时查看提交记录，只有以前的两次提交了
## d256c95bd173362053eb04bea4ff91a5865fc8f2 add number
## a2989999cd2f7aac717fde3b5f7354f3c9deba6d first commit

$ git reflog                   # 此时如果想回到 ’modify third‘，可以使用此命令找到 commit id
## d256c95 HEAD@{0}: reset: moving to d256c95bd173362053eb04bea4ff91a5865fc8f2
## 2fa86e3 HEAD@{1}: commit: modify third
## d256c95 HEAD@{2}: commit: add number
## a298999 HEAD@{3}: commit (initial): first commit

$ git reset --hard 2fa86e3      # 我们又回到最新的版本了

3. 撤销操作

## 场景1：修改了文件未暂存，想直接丢弃修改: git checkout -- file
## 场景2：修改了文件还添加到了暂存区，想丢弃修改分两步，第一步: git reset HEAD file，就回到了场景1，第二步按场景1操作。
## 场景3：已经提交了不合适的修改到版本库时，想要撤销本次提交，使用版本回退，不过前提是没有推送到远程库。

$ git checkout -- version.txt   # 场景1

$ git reset 2fa86e3d1416ddcfca369ccec5ef8392156e3256 version.txt # 场景2
$ git checkout -- version.txt

4. 远程仓库

## 先使用 Github 账户的邮箱创建 RSA 密钥对(什么都不填，一直回车)
$ ssh-keygen -t rsa -C "jiwanger@126.com"

## 登录 Github 账户，在设置中 SSH abd GPG keys -> New SSH key 添加上一步创建的公钥

## 方式一: 使用 git/https 协议克隆远程库(如果是自己的项目，那么具有修改和提交的权力)
$ git clone git@github.com:jiwanger/example.git
$ git clone https://github.com/jiwanger/example.git

## 方式二: 添加远程库 
$ git remote add origin git@github.com:jiwanger/example.git
$ git remote add origin https://github.com/jiwanger/example.git

## -u 参数，Git 不但会把本地的 master 推送的远程 master 分支，还会关联起来
$ git push -u origin master 	# 首次推送
$ git push origin master        # 以后推送

5. 分支管理


$ git checkout -b dev            # 创建一个 dev 分支并切换（git branch dev & git checkout dev） 
$ git switch -c dev              # 同上，新版的 Git 使用 switch 更科学
$ git branch                     # 查看分支 
$ git branch -a                  # 查看所有分支，包括远程分支 
$ git checkout master            # 切换到 master 分支
$ git switch master              # 同上，新版的 Git 使用 switch 更科学

$ git merge dev                  # 将 dev 分支合并到当前分支
$ git branch -d dev              # 删除 dev 分支

6. 冲突解决 -- 解决冲突就是把 Git 合并失败的文件手动编辑为我们希望的内容，再提交。

$ git switch -c dev        # 创建并切换到 dev 分支，此时修改某一个文件并提交
$ git switch master        # 回到 master 分支，修改同一个文件并提交
$ git merge dev            # 尝试合并 dev 到 master 分支，会提示冲突 

## 合并时也可以禁用 Fast forward 模式，这样在合并后的历史有分支，能看出来曾经做过合并
$ git merge --no-ff -m "merge with no-ff" dev  # 

$ git status                       # 状态显示冲突
$ git commit -m "conflict fixed"   # 冲突的文件修改后再提交
$ git log --graph --pretty=oneline # 查看合并的情况

$ git branch -d dev                # 合并分支后删除 dev 分支
$ git branch -d dev                # 合并分支后删除 dev 分支

7. Bug 分支 -- 在 dev 分支做开发时，碰到 master 分支有 Bug 需要解决

$ git stash                   # (dev 分支) 如果在当前分支还不想 commit 时，可以选择 stash
$ git checkout master         # 切换到主分支（去处理主分支上的 Bug）

$ git checkout -b issue-101   # 创建 Bug 分支（修改 Bug 并提交更改）

$ git switch master           # 回到主分支
$ git merge --no-ff -m "merged bug fix 101" issue-101   # 合并 issue-101
$ git branch -d issue-101     # 删除 issue-101 分支

$ git switch dev              # 回到 dev 分支继续工作
$ git stash pop               # 恢复存储的工作(可能需要解决冲突)
## 如果保存有多个（执行了多次 git stash）
$ git stash list              # 查看存储的列表
$ git stash apply {stash}     # 恢复指定的现场

## 在 master 分支上修复了 bug 后，发现 dev 分支是从 master 分支分出来的，所以，这个 bug 其实在当前 dev 分支上也存在
$ git cherry-pick 4c805e2     # 在 dev 分支上只需要把 fix bug 101 这个提交所做的修改“复制”到 dev 分支

8. 多人协作

$ git remote -v 		 # 抓取和推送的 origin 的地址。如果没有推送权限，就看不到 push 的地址
$ git push origin master # 推送指定分支(master)到远程库对应的远程分支 
$ git push origin dev    # 推送 dev 分支(如果失败：$ git pull 先把远程分支取下来解决冲突)

$ git checkout -b nate origin/nate         # 本地创建和远程分支对应的分支
$ git branch --set-upstream dev origin/dev # 建立本地分支和远程分支的关联

$ git rebase             # 可以把本地未 push 的分叉提交历史整理成直线(提交变得好看)

9. 标签管理

$ git switch dev         # 切换到需要打标签的分支 
$ git tag v0.4           # 打标签
$ git tag                # 产看当前分支上的标签（字母顺序）

$ git log --pretty=oneline --abbrev-commit  # 查看提交记录
$ git tag v0.2 f52c633                      # 将标签打在指定的 commit id 上

$ git show v0.4          # 查看标签信息
$ git tag -a v0.1 -m "version 0.1 released" 1094adb  # 带说明的 tag, -a 指定标签名，-m 指定说明文字

$ git push origin v0.4   # 推送标签到远程(创建的标签存储在本地)
$ git push origin --tags # 一次性推送全部尚未推送到远程的本地标签

$ git tag -d v0.2                 # 删除标签
$ git push origin :refs/tags/v0.2 # 标签已经推送到远程，先从本地删除，再删除远程
